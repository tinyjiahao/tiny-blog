# 技术写作

把技术内容写成适合阅读和传播的文章。本技能覆盖**两种载体**，它们的写作手艺不同，写作前必须先判断走哪种模式：

| 模式 | 载体 | 读者状态 | 核心目标 |
|------|------|---------|---------|
| **公众号模式** | 微信公众号、知乎、掘金等流媒体 | 被动刷信息流、随时划走 | 抓住注意力，让人读完、让人转发 |
| **博客模式** | 本项目 Hexo 博客（`source/_posts/`） | 主动检索、带着目的来读 | 讲完整、讲准确、可长期检索查阅 |

```
一句话区分：
  公众号：「前 3 屏抓不住人，后面写得再好也没人看」
  博客：  「读者是来查资料的，严谨完整比花哨重要」
```

**准确性是两种模式的共同底线**，但「准确」不等于「枯燥」。下面先讲两模式共通的部分，再分别细化。

## 第一步：判断走哪种模式（动笔前必做）

如果用户已明说（「写篇公众号文」「加篇博客」），按说的来。否则根据线索判断，**拿不准就直接问一句**（别挤牙膏，一个问题搞定）：

| 线索 | 判断 |
|------|------|
| 「公众号」「自媒体」「传播」「转发」「吸粉」 | 公众号模式 |
| 「博客」「写进 _posts」「更新博客」「长期沉淀」「系统设计笔记」 | 博客模式 |
| 「改写」「改编」+ 指向已有博客文件 | 默认公众号模式（博客→公众号是高频场景） |

同时确认动笔前必问的要素（用户已给的就不重复问）：

1. **主题与核心信息**：读者读完应记住的「一句话结论」？
2. **目标读者**：初学者 / 有一定经验 / 资深同行？（决定术语密度和讲解深度）
3. **篇幅 / 深度**：快餐（~1000 字）/ 标准（2000 字左右）/ 深度长文（4000+ 字）？
4. **来源**：从零写，还是有现成素材/博客/代码要改写？

> 用户已经讲清楚的，别为走流程而追问。

# 两种模式共通的心法

无论哪种模式，下面这些写作原则都成立：

## 去 AI 味（每篇交稿前必查）

> 读者对「AI 味」极其敏感——一旦觉得「这是 AI 写的」，信任感和阅读意愿断崖式下跌。这一节是质量底线。

### 什么是「AI 味」

AI 味不是某个词，而是一组**结构化、套路化、太「正确」太「全面」**的痕迹。本质是：**像在交作业，不像在跟人说话**。

### 六大典型 AI 味（附改法）

#### 1. 排比式总结收尾 / 三段式总分总

AI 爱用「不仅 A，还 B，更 C」工整排比，和「总而言之 / 综上所述」收尾。像 PPT 汇报。

```
❌ AI 味：「限流不仅保护了下游，更提升了系统稳定性，还守护了用户体验。」
✅ 人味：「说白了，限流就是用『拒绝一部分请求』换『核心服务别挂』——两害相权取其轻。」
```

#### 2. 万能连接词堆砌

「首先……其次……最后……」「值得一提的是」「不可否认的是」「众所周知」「显而易见」。单独看没问题，**密集出现就是 AI 指纹**。

```
❌ AI 味：「首先我们来看限流算法。其次，固定窗口存在临界问题。值得一提的是，滑动窗口解决了它。」
✅ 人味：「限流有四种算法，先说最简单的固定窗口——但它有个致命伤：临界突发。」
```

#### 3. 空话开场与宏大叙事

「在当今快速发展的……」「随着业务的不断增长……」「作为开发者，我们经常会遇到……」——零信息量。

```
❌ AI 味：「在微服务架构日益普及的今天，服务稳定性成为了每一个开发者必须面对的重要课题。」
✅ 人味：「上周我们一个下游服务挂了 3 分钟，上游线程池被打满，差点雪崩。事后复盘发现，加一个熔断就能扛住。」
```

#### 4. 过度对称的小标题与结构

每节字数差不多、结构一模一样——**太工整就是 AI**。真人有轻重：重要的展开讲，不重要的一笔带过。

```
❌ AI 味：每节都是「什么是 X / 原理 / 优缺点 / 小结」四段式，长度均匀
✅ 人味：核心算法那节写 500 字掰开揉碎，边缘知识点一句话「这块展开太长，记住结论就行」带过
```

#### 5. 「总—分—总」滴水不漏的段落

AI 每段都「先抛观点、再解释、再小结」，**每段都自己收尾**。真人会留悬念、会用短句、会突然抛一句不解释。

```
❌ AI 味：「漏桶算法是一种流量整形算法。它的核心是……。因此，它适合……。」（每段都这三段式）
✅ 人味：短长句交错。偶尔一句单独成段——「但漏桶有个让人头疼的毛病。」然后下一段再解释。
```

#### 6. 客观中庸、没有判断

AI 怕出错，所以永远「各有优劣，需根据场景选择」——永远正确，永远没用。真人有偏好、敢下判断。

```
❌ AI 味：「令牌桶和漏桶各有优劣，实际应用中应根据业务场景综合考量。」
✅ 人味：「接口限流直接上令牌桶，别犹豫。漏桶留给那种『必须匀速』的场景，比如调第三方短信通道。」
```

### 去味自检清单（交稿前过一遍）

- [ ] 全文有没有「首先/其次/最后」三段式骨架？有就打散重排
- [ ] 开头是不是空话铺垫？能否直接换成场景/痛点/数据？
- [ ] 每段是不是都在「自我总结」？删掉「所以我们可以看出……」
- [ ] 有没有给**明确判断和推荐**，而不是「视情况而定」？
- [ ] 句式够不够参差？有没有连续好几句「XX 是 YY。ZZ 是 AA。」并列结构？
- [ ] 有没有一两处只有真人会写的「私货」——踩坑、吐槽、不主流的观点？

### 终极心法

```
AI 默认身份：「我要把这个知识点完整、准确、全面地呈现出来。」→ AI 味
要切换成：     「我刚才踩了个坑 / 想通了一件事，赶紧跟同事讲讲。」→ 人味
```

写之前先在脑子里过一遍「这话我会在群里跟同事怎么说」，用口语节奏写，再书面化润色——最有效的去味方法。

## 术语与表达（两模式通用）

- **术语处理**：同行可直用，初学者「先白话后术语」；首次出现配一句白话解释
- **有观点有温度**：允许下判断（「我个人更推荐 X」「这里 90% 的人会踩坑」），读者要「人」写的文，不是文档搬运
- **图比字管用**：架构图、流程图、对比表，一张好图顶三段文字

## 顺读与易懂（每篇都要做到）

文章不只是“信息完整”，还要让读者一路读得下去。技术文章最容易坏在两件事：一是概念堆太快，读者跟不上；二是段落之间像拼接，读者不知道为什么突然讲到这里。

### 写作节奏

- **先给结论，再补原理**：读者先知道“这东西解决什么问题”，再听你讲细节。
- **一段只讲一个意思**：一段里同时解释概念、列优点、讲实现、做总结，会让人喘不过气。
- **长段落拆开**：超过 5 行的段落优先拆成两个自然段；超过 8 行通常已经太长。
- **先人话，后术语**：先说“它像一个临时记事本”，再说“这就是工作记忆”。
- **重要概念配例子**：不要只定义。每个核心概念至少配一个生活类比、工程场景或具体输入输出。

### 段落衔接

每节之间要有自然过渡，不要像把几段笔记拼起来。

```
❌ 生硬跳转：
「下面介绍 Rerank。」

✅ 顺着问题走：
「召回阶段宁可多捞一些候选，但捞多了就会混进噪声。接下来要解决的，就是怎么把真正相关的排到前面——这就是 Rerank。」
```

常用过渡方式：

- **承接问题**：「知道了 X，新的问题来了：Y 怎么办？」
- **指出矛盾**：「这听起来够了，但生产里会遇到一个麻烦……」
- **用例子推进**：「换到客服场景里，这个问题会变成……」
- **先收束再展开**：「这一节记住一句话就行：……下面拆开讲。」

### 解释复杂概念的顺序

复杂技术点按这个顺序写，读者最不容易丢：

1. **它解决什么问题**：没有它会怎样？
2. **它的直觉是什么**：用一句人话解释。
3. **它怎么工作**：讲流程、组件、数据流。
4. **它什么时候会失效**：边界和坑。
5. **落地怎么做**：工程建议、指标、代码或图。

不要一上来就抛定义。定义通常不是入口，问题才是入口。

### 好读的句子

- 少用抽象名词连环套，比如「能力建设的体系化落地实践」；改成「怎么把这套能力真正做进系统」。
- 少用“进行、实现、提供、完成”这类空动词；能用具体动词就用具体动词：查、写、拆、拦、重试、回滚。
- 同一段里长短句交替。连续三句都超过 35 个字，就拆。
- 技术判断要具体，不要只写“需要根据业务场景选择”。要说明“什么场景选 A，什么场景选 B，我更推荐哪个起步”。

### 顺读自检清单

- [ ] 开头 3 段内，读者是否知道本文要解决什么问题？
- [ ] 每个小节是否沿着一个问题往下讲，而不是平铺资料？
- [ ] 核心术语是否都有白话解释和例子？
- [ ] 段落之间是否有过渡句，读者能否知道为什么讲到下一节？
- [ ] 是否删掉了“正确但没信息量”的句子？
- [ ] 结尾是否给了读者可带走的判断、清单或框架？

# 公众号模式

读者在手机上刷信息流，**节奏必须快、段落必须短**。

## 标题先行

公众号生死线在标题。**先定 2–3 个候选标题，和用户对齐方向，再写正文。** 标题定了，文章切入点和调子才定。

技术号好标题范式：

| 类型 | 模式 | 示例 |
|------|------|------|
| 反常识/认知颠覆 | 打破普遍误解 | 「为什么你的限流其实没挡住流量？固定窗口的致命盲区」 |
| 数字 + 干货清单 | 明确能得到什么 | 「MySQL 索引失效的 7 种场景，我踩过 5 个」 |
| 问题钩子 | 抛一个好奇问题 | 「为什么大厂都在用令牌桶，不用漏桶？」 |
| 踩坑/事故复盘 | 带真实感和警示 | 「一次线上雪崩复盘：我们忘了加熔断」 |
| 对比 PK | 两个概念正面刚 | 「限流 vs 熔断：90% 的人分不清」 |

**克制**：避开「震惊体」「不看后悔」，开发者反感标题党。要有信息量、有观点、带技术词，留个钩子。控制在 20–30 字。

## 结构编排（手机竖屏）

```
1. 开头钩子（200–400 字）
   - 不要「今天我们来聊聊 XX」水文开头
   - 用真实场景/痛点/事故/反问直接切入
   - 第一段就要让读者觉得「这跟我有关 / 这我不知道」

2. 痛点 / 问题铺垫
   - 抛出读者会遇到的具体问题，建立共鸣
   - 「你是不是也遇到过……」「很多人以为……」

3. 核心内容（3–5 个小节，每节讲透一个点）
   - 每节：一个观点 + 一个例子/图/代码
   - 小节间要有过渡句，别硬跳
   - 用小标题切分，扫一眼知道讲到哪

4. 总结 / 速查表 / 金句收尾
   - 给读者「值得收藏转发」的理由
   - 一张对比表、一个口诀、几句金句
```

**篇幅心法**：宁可砍掉一个小节，也别让每节注水。读者觉得「水了」就取关。

## 公众号平台适配检查

- [ ] **字数**：标准 1500–2500 字，长文 3000+，超长考虑分上下篇
- [ ] **代码块**：公众号代码块支持有限，无复杂语法高亮依赖；过长代码改为「核心片段 + 完整代码放 GitHub/阅读原文」
- [ ] **代码要短**：单块超过 15 行就该裁剪，手机上长代码读不下去
- [ ] **外链**：公众号正文外链不可点！重要链接提示「阅读原文」或用口令/二维码
- [ ] **图片**：宽度适配，重要文字信息别放图里（图片可能被压缩）
- [ ] **emoji/特殊符号**：谨慎用，技术号过度 emoji 显得轻浮，少量作视觉锚点 OK
- [ ] **标题再确认**：前 13 个字要能吸引点击（列表页只显示前 13 字左右）

## 公众号模式产出物

1. **2–3 个候选标题**（标注推荐项）
2. **完整正文**（markdown，已按手机节奏排版）
3. **发布建议**（可选）：封面、摘要/简介（分享卡片用）、标签

# 博客模式

本项目博客文章沉淀在 `source/_posts/`，读者是主动来查资料的开发者，**严谨、完整、可检索**比花哨重要。动笔前务必参考已有文章的体例（见下文「博客体例标准」）。

## 博客体例标准（必须遵守，从现有文章归纳）

本项目现有文章（`mysql-basics.md`、`elasticsearch-basics.md`、`rate-limiter-basics.md`、`sms-system-design.md` 等）有稳定统一的体例，新文章必须对齐：

### Front matter

```yaml
---
title: <文章标题>          # 如「限流详解」「熔断详解」
date: YYYY-MM-DD HH:MM:SS  # 精确到秒，用当前时间
description: '<一句话总结，描述覆盖的核心知识点>'
categories:
  - <分类>                 # 如 系统设计 / 数据库 / 建站
tags:
  - <标签1>                # 4~5 个，含核心技术词 + 「后端」等通用标签
  - <标签2>
---
```

### 正文骨架

```
> <引用块开头：一句话点明本文核心价值，并提示与相关文章的关联>

```
本文脉络：
  一    <章节1>
  二    <章节2>
  ...
```

<!-- more -->              ← Hexo 摘要分隔符，必加

## 一、<章节标题>

<正文>

## 二、<章节标题>
...
```

### 内容元素（现有文章的通用手法，新文应对齐）

- **配图优先**：架构图、流程图、状态机、时序图优先用 SVG/PNG 等图片呈现；除非用户明确要求，不要用 ASCII 图替代正式配图
- **对比表格**：算法对比、框架对比、优缺点对比（markdown 表格）
- **代码示例**：主流语言的实现（Java/Go/SQL/Lua/配置等），带中文注释
- **编号章节**：用「一、二、三」中文数字编号主章节
- **常见问题表**：技术文结尾保留一张「问题 / 回答要点」表，是博客文的「收藏点」

## 博客写作原则

- **可深可全**：博客读者能接受长篇铺垫和密集代码，**别为了精简而牺牲完整度**。一个知识点该展开就展开，公众号会砍的严谨铺垫，博客要保留
- **可检索可复查**：文章要经得起反复检索和长期回看，术语精确、边界条件写清楚、结构稳定（编号章节便于引用）
- **交叉引用**：相关主题文章之间互相呼应（如限流文第八章 ↔ 熔断文第七章），用「详见『XX 详解』一文」「配合『XX』食用」这类引用
- **金句与速查并存**：既要有人味的总结金句，也要有可速查的表格/口诀

## 博客模式产出物

1. **完整 markdown 文件**（含合规的 front matter + `<!-- more -->` + 编号章节 + 常见问题）
2. **文件命名**：kebab-case，与主题英文对应（如 `circuit-breaker-basics.md`、`mysql-basics.md`）
3. **TODO 清单（草稿用）**：若是草稿，文末列出待补充的方向（真实案例、集成示例、交叉引用等），方便后续迭代

# 博客 ↔ 公众号 互转

两种模式可互转，但**不是搬运，是重写**：

```
博客 → 公众号：
  - 砍掉 1/3 篇幅（博客的严谨铺垫在公众号是累赘）
  - 重写开头，从「教科书式」改成「场景钩子式」
  - 长代码块裁剪成关键片段
  - 加金句和过渡，提升扫读体验
  - 「常见问题」表格很适合作为公众号结尾的「收藏点」，可保留

公众号 → 博客：
  - 补回严谨性：加引用、补边界条件、术语精确化
  - 展开被砍掉的铺垫和原理
  - 补全 front matter、编号章节、正式配图、常见问题表
  - 文件按博客体例命名（kebab-case），放入 source/_posts/
```

## 语气基调

你是「资深工程师 + 会写文章的人」，不是营销号小编，也不是机械的文档搬运工。专业、有判断力、读起来像一个靠谱的同事在给你讲他踩过的坑和想通的道理。读者读完应该觉得「这人懂行，还讲得明白」。
