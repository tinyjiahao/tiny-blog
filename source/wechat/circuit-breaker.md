# 一次线上雪崩复盘：我们忘了给下游加熔断

> 原文发表于个人博客「熔断详解」，本文是面向公众号的改写版。技术细节完整版见博客原文。

上周三晚高峰，我们一个下游服务卡了 3 分钟。

就这 3 分钟，上游订单服务的线程池被打满，请求开始排队超时；订单服务一卡，用户中心调它也跟着超时……整条调用链差点雪崩。事后复盘，根因就一句话：**我们忘了在调用下游的地方加熔断**。

下游抖一下，凭什么能把上游拖垮？今天就从这次事故聊聊熔断——这个高可用系统最该有、却最常被漏掉的兜底。

## 一、下游卡 3 分钟，为什么能拖垮整条链？

很多人以为，下游挂了，上游顶多报错。但真正的杀手不是「挂」，是「**变慢**」。

```
调用链：用户中心 → 订单服务 → 支付服务 → 银行网关

银行网关响应从 50ms 飙到 5s：
  ↓
支付服务线程全在等银行网关 → 线程池耗尽
  ↓
订单服务调支付服务也开始超时 → 订单线程池耗尽
  ↓
一路向上传染，全站不可用
```

注意，下游没挂，它只是**慢**。但上游每次调用都会傻等一个超时时间（比如 3 秒）才放弃，这几秒里，宝贵的线程一直被占着。流量一进来，线程池瞬间见底。

这时候你可能想：我设了超时啊，也加了重试啊，怎么还崩？

## 二、超时和重试，为什么救不了你

| 手段 | 作用 | 致命局限 |
|------|------|---------|
| **超时** | 避免无限等待 | 等待期间线程照样占着；下游持续故障时，每次都得等满超时才释放 |
| **重试** | 应对瞬时抖动 | 下游一直坏，重试反而**把流量放大 2~3 倍**，加速雪崩 |

根本问题在于：**超时和重试都是「没记忆」的。**

它们每次调用都独立判断——这次失败了，下次还是照常去调。明明下游已经连续失败 100 次了，上游还是执着地去试，每次都傻等一个超时。

这就像你知道同事请假了，还每隔 5 分钟去敲一次门，每次敲 3 分钟没人开才走。**熔断干的事，就是给你加上「记忆」**：连续失败到一定程度，就认定下游「病了」，在一段时间内直接快速失败，不再去打扰它。

一句话：**熔断是给调用加上记忆，让它别再当舔狗。**

## 三、熔断长什么样：三个状态

熔断器是个状态机，三个状态，类比家里的保险丝：

```
                    失败率达阈值
   ┌────────────┐ ─────────────────► ┌────────────┐
   │  Closed    │                    │   Open     │
   │  正常放行   │ ◄───────────────── │  全部快失败 │
   └────────────┘   半开探测成功      └─────┬──────┘
         ▲                                  │
         │                          等冷却时间结束
         │                                  ▼
         │                           ┌────────────┐
         └───────────────────────── │ Half-Open  │
                                     │ 放几个试探   │
                                     └────────────┘
```

| 状态 | 干什么 |
|------|--------|
| **Closed（关闭）** | 正常工作，默默统计失败率 |
| **Open（打开）** | **所有请求直接快速失败**，不去调下游，给它恢复时间 |
| **Half-Open（半开）** | 放几个请求试探，成功就回 Closed，失败就继续 Open |

这里有个最常被答错的点：**Open 状态是 100% 全拒，不是「按比例拒绝」**。

很多人把熔断和限流搞混——限流才是「按比例放行」（看流量），熔断 Open 是「全部拒绝」（看下游健康度）。你不需要纠结「拒多少」，Open 期内一个都不放过去。

**为什么要有个 Half-Open？** 直接从 Open 切回 Closed 不行吗？不行。万一下游还没真恢复，流量一涌进去又把它压垮了。Half-Open 就是「灰度恢复」——先放几个请求探探路，活了再全放开。

## 四、什么时候该跳闸

熔断打开的依据，是统计窗口里的「下游健康度」：

- **错误率**（最常用）：失败占比超阈值就跳，比如 50%
- **慢调用率**：超时请求占比超阈值，专门对付「慢但不错」的假死
- **失败次数**：绝对失败数超阈值

有个细节容易被忽略：**一定要设「最小请求数」**。

如果窗口里只有 2 个请求还都失败了，错误率 100% 看着吓人，其实只是样本太少。一般设最小 20 个请求才开始统计，避免系统刚启动时的误判。

## 五、跳闸之后，给用户什么

熔断打开，请求会快速失败。但**对用户的感受，取决于你的降级逻辑**。

没有降级的熔断 = 直接报错，用户看到白屏，体验灾难。所以熔断一定要配降级：

- **读接口**：返回缓存、默认值、热门推荐，别给空白
- **列表查询**：返回首屏固定数据，藏掉非核心模块
- **写接口**：千万不能假装成功！转成异步，提示「处理中」
- **非核心功能**（推荐、评论）：直接隐藏，保住主流程

> 写操作是最容易踩坑的。下单接口熔断了，你返回「下单成功」，结果订单根本没建——这种「假成功」比报错可怕十倍。正确做法是写进 MQ，告诉用户「正在处理」，异步再补。

## 六、熔断和重试，天生一对冤家

这一节是事故复盘里最关键的一条。

```
上游重试 + 下游故障 = 雪崩加速器

上游调下游失败 → 重试（流量 ×2）
  → 下游更扛不住 → 错误率更高
  → 如果没有熔断，上游继续重试 → 流量持续放大
  → 下游彻底压垮，上游也跟着死
```

**熔断在这里救场**：错误率飙到阈值，熔断一打开，重试就被抑制了——直接走降级，不再去调下游。

所以记住这个铁律：**重试必须配合熔断**。只有重试没有熔断，故障时重试就是帮凶；熔断打开期间，重试应当被压制。重试本身也要有退避（backoff）和次数上限，不能无限重试。

## 七、用什么框架

不用自己造轮子，三个主流选择：

| 框架 | 一句话评价 | 还能用吗 |
|------|----------|---------|
| **Hystrix** | Netflix 经典，熔断 + 线程池隔离的思想奠基者 | 2018 年起停更，老项目才有 |
| **Resilience4j** | Hystrix 的现代继任者，函数式、轻量 | **新项目首选** |
| **Sentinel** | 阿里出品，熔断+限流一体，带可视化 Dashboard | 国内 Java 主流 |

以 Resilience4j 为例，配置就这么几行：

```java
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)                          // 错误率 50% 跳闸
    .slowCallRateThreshold(60)                         // 慢调用率 60% 跳闸
    .minimumNumberOfCalls(20)                          // 至少 20 个请求才统计
    .waitDurationInOpenState(Duration.ofSeconds(10))   // Open 持续 10s
    .build();
```

配置项看着多，翻译成人话就是：**「至少凑够 20 个请求，错误率到一半、或慢调用到六成，就跳闸歇 10 秒」**。

## 八、我们的事故，复盘后改了什么

回到开头那次事故。复盘后我们做了三件事，正好对应熔断落地最该盯的几个点：

1. **给所有调外部/下游的入口加熔断**。纯本地内存操作不需要，但调第三方 API、网关、下游服务的，一个都不能少。
2. **熔断 + 降级 + 告警 三件套**。熔断触发就自动降级保体验，同时告警让人介入——光熔断不告警，故障被「掩盖」了反而更危险。
3. **参数能动态调**。生产环境的阈值、冷却时间，得支持不重启就改（Sentinel Dashboard 这点很方便）。压测时的参数和真实流量下的参数，往往差很多。

## 九、一张表收尾

最后用一张表把这次复盘的核心收一下，方便收藏：

| 你可能想问 | 一句话答案 |
|----------|----------|
| 下游变慢为什么能拖垮上游？ | 上游线程傻等超时，被一个个占满，整条链传染 |
| 超时+重试够不够？ | 不够，它们没记忆，故障时重试还会放大流量 |
| 熔断 Open 是按比例拒吗？ | 不是，是 100% 全拒；按比例拒的是限流 |
| 为什么要 Half-Open？ | 灰度恢复，确认下游真活了再放全量 |
| 跳闸后给用户什么？ | 读返回缓存/默认值，写转异步，别假装成功 |
| 重试和熔断什么关系？ | 冤家，熔断能压制重试；只重试不熔断会雪崩 |
| 用什么框架？ | 新项目 Resilience4j，国内 Java 用 Sentinel |

---

**一句话总结这次复盘**：超时和重试只能对付瞬时抖动，**真正防雪崩的，是熔断那份「记忆」**。

如果你也在维护有外部依赖的服务，花十分钟检查一下：该加熔断的地方，都加了吗？
